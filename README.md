# Velocity API

A RESTful backend API for the [Velocity Kanban board](https://github.com/Scuuff/velocity-kanban). Built with **Node.js + Express + MongoDB Atlas**.

> Built as Project 2 of a full-stack internship program — focused on API design, server-side logic, input validation, and persistent storage.

🔗 **Live API:** _add link after deploying to Render_
🎨 **Frontend:** [velocity-kanban](https://scuuff.github.io/velocity-kanban/) *(currently uses localStorage; will be wired to this API in a follow-up)*

---

## Features

- **Full REST CRUD** for cards: list, get, create, update, delete
- **MongoDB Atlas** cloud persistence — survives server restarts and is accessible from anywhere
- **Mongoose schema validation** — enum constraints on column / priority, date format checks
- **Layered input validation** — middleware checks request bodies *before* hitting the database, returns clear 400 errors with detailed field-level messages
- **Proper REST status codes** — 200 / 201 / 204 / 400 / 404 / 500
- **CORS enabled** so the browser frontend can call the API from any origin
- **Centralized error handling** with a single Express error middleware
- **Environment-based config** — connection string in `.env`, never committed
- **Filterable list** — `GET /api/cards?column=todo`

## API endpoints

### Cards

| Method | URL | Description | Body | Returns |
|---|---|---|---|---|
| `GET` | `/` | Health check | — | `200` `{ name, status, docs }` |
| `GET` | `/api/cards` | List all cards (filter by `?column=todo\|doing\|done`) | — | `200` `{ count, cards: [...] }` |
| `GET` | `/api/cards/:id` | Get one card by id | — | `200` card / `404` not found |
| `POST` | `/api/cards` | Create a card | `{ text, column?, priority?, dueDate?, subtasks? }` | `201` created card |
| `PUT` | `/api/cards/:id` | Partial update of a card | any subset of the above fields | `200` updated card |
| `DELETE` | `/api/cards/:id` | Delete a card | — | `204 No Content` |

### Events (activity log)

| Method | URL | Description | Returns |
|---|---|---|---|
| `GET` | `/api/events?limit=20` | Recent activity events, newest first. Auto-logged by the cards controller on every CRUD operation. Types: `created`, `moved`, `updated`, `subtask_toggled`, `deleted`. | `200` `{ count, events: [...] }` |

### Stats

| Method | URL | Description | Returns |
|---|---|---|---|
| `GET` | `/api/stats/completions?days=14` | Daily completion counts for cards **currently in Done**, bucketed by the day they most recently became done. Backfills missing days with `0`. | `200` `{ days, total, max, series: [{ date, weekday, count }] }` |

### Card schema

```js
{
  _id:         ObjectId,         // auto-generated by MongoDB
  text:        String,           // required, max 200 chars
  column:      'todo' | 'doing' | 'done',     // default: 'todo'
  priority:    'easy' | 'medium' | 'hard',    // default: 'medium'
  dueDate:     String,           // 'YYYY-MM-DD' or '' (empty)
  subtasks: [                    // array of subtask objects
    {
      _id:  ObjectId,            // auto-generated per subtask
      text: String,              // required, max 120 chars
      done: Boolean              // default: false
    }
  ],
  createdAt:   Date,             // auto-managed by Mongoose
  updatedAt:   Date              // auto-managed by Mongoose
}
```

### Example requests

```bash
# List all cards
curl http://localhost:3000/api/cards

# Filter by column
curl http://localhost:3000/api/cards?column=doing

# Create a card
curl -X POST http://localhost:3000/api/cards \
  -H "Content-Type: application/json" \
  -d '{"text":"Ship the API","priority":"hard","column":"doing"}'

# Update a card (partial — only fields you send are changed)
curl -X PUT http://localhost:3000/api/cards/<id> \
  -H "Content-Type: application/json" \
  -d '{"column":"done"}'

# Delete a card
curl -X DELETE http://localhost:3000/api/cards/<id>
```

### Error responses

All errors return JSON with an `error` field. Validation failures include a `details` array:

```json
{
  "error": "Validation failed",
  "details": [
    "\"text\" is required and must be a non-empty string",
    "\"priority\" must be one of: easy, medium, hard"
  ]
}
```

## Getting started

### 1. Clone and install

```bash
git clone https://github.com/Scuuff/velocity-api.git
cd velocity-api
npm install
```

### 2. Set up MongoDB Atlas (free)

1. Sign up at https://www.mongodb.com/cloud/atlas/register
2. Create a free **M0 cluster**
3. Create a database user and add your IP to the access list
4. Copy the connection string

### 3. Configure environment

```bash
cp .env.example .env
# Then edit .env and paste your MongoDB connection string
```

`.env` example:
```
PORT=3000
MONGODB_URI=mongodb+srv://user:password@cluster.xxxxx.mongodb.net/velocity?retryWrites=true&w=majority
```

### 4. Run

```bash
npm start              # production mode
# or
npm run dev            # nodemon — auto-restarts on file changes
```

Server starts on `http://localhost:3000`. You should see:

```
✓ MongoDB connected
✓ velocity-api running on http://localhost:3000
```

## Architecture notes

A few design decisions worth highlighting:

- **Two-layer validation** — application-level (in `middleware/validate.js`) runs *before* the database, catching bad requests cheaply. Schema-level (in `models/Card.js`) is the final safety net at the database boundary. Defense in depth.
- **Separate validators for create vs update** — `validateCardBody` requires `text`; `validateCardUpdateBody` makes every field optional but rejects empty bodies. Both share field rules but with different "required" semantics.
- **Partial updates with `findByIdAndUpdate`** — the controller builds the update object explicitly so `undefined` fields aren't accidentally written. Mongoose's `runValidators: true` ensures schema constraints still apply on updates.
- **Status codes match REST conventions** — `201` for create, `204` for delete (no body), `200` for update/get, `400` for client error, `404` for not found, `500` for server crash. Makes the API self-documenting.
- **Centralized error handler** in `server.js` catches any error from a controller's `next(err)` call. Controllers can throw or call `next(err)` without each one repeating its own try/catch boilerplate.
- **Validates ObjectId format before hitting DB** — `mongoose.isValidObjectId(id)` rejects garbage like `/api/cards/notarealid` with a fast `400` instead of letting Mongoose throw a CastError later.

## What I learned

- **REST conventions matter.** Returning `204 No Content` for DELETE feels weird at first (no body!) but is exactly correct — and clients can rely on it.
- **Validation is layered, not centralized.** Catch obvious garbage in middleware, let the schema do final integrity checks. Each layer has a different job.
- **Mongoose is a small ORM with big quirks.** `findByIdAndUpdate` doesn't run validators by default — you have to opt in with `runValidators: true`. `findById` returns `null` for not found, but the `:id` route param can also be a malformed string. Two different failure modes, both need handling.
- **`.env` files are gospel.** Never commit secrets. The connection string in this project's `.env` contains a real database password, and it's gitignored for that reason.
- **CORS is not optional for browser clients.** Without `app.use(cors())`, the Kanban frontend running on `localhost:5500` couldn't call this API at all — same-origin policy would block it.
